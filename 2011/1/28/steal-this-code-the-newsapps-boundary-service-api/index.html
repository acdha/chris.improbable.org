Steal this code: The Newsapps Boundary Service API!

@coloneltribune's Voting District (#42053)

For the past two months I’ve been working at fever pitch on a side-project that I’m very excited about. My enthusiasm infected newfound Pythonista and honorary apps developer Ryan Nagle and together we’ve hammered into shape a project that I hope a lot of news application developers, open government geeks and others will get really excited about: the Newsapps Boundary Service API, including a public implementation for Illinois.

What is it?

Cribbing from the README:

The Boundary Service is a ready-to-deploy system for aggregating regional boundary data (from shapefiles) and republishing that data via a RESTful JSON API. It includes an extensive demo site which demonstrates the functionality of the API using client-side javascript.

This project is aimed at providing a simple service for newsrooms, open-government hackers and others to centralize and build on regional GIS data. You can see the instance we’ve configured for Chicago & Illinois, along with much more detailed information about the API at http://boundaries.tribapps.com/.

Why do we need this?

I’ve learned a huge number of things about GIS since starting at the Tribune 14 months ago. Its become one of my favorite subjects to hack on, but also one of the ones I’m most loathe to re-implement for every project. A lot of things about GIS are hard and its a shame that with each project we build we end up writing another shapefile loader. The Boundary Service provides us an endpoint to access the geographic data we need without constantly rebuilding the infrastructure for it.

The service has already been put to use powering the Ward Finder and individual Ward maps on the Chicago Tribune Election Center.

What should I do with it?

Its my fervent hope that other newsrooms will deploy the API for their own geographies. Ideally those implementations will be public, but even if it only serves to raise the geographic IQ within the newsroom it will be a success. I’m also hoping developers will pick up the project and build on top of it. Adding KML serialization to the service would be a very high-value feature, as would a more throughly developed front-end administrative interface.

What else? You tell us! Leave your feedback and ideas in the comments.
